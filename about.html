About the project

Making the models and website

This page explains the main problems I faced when modelling three retro handheld consoles — the Nintendo Game Boy (1989), Game Boy Advance SP (2003) and Sony PlayStation Portable (2004) — and connecting them to an interactive Three.js museum website. The focus is on practical modelling, export and Three.js integration issues.

Process and tools

How I made the three models

I created all three handheld console models in Blender from scratch. I started by blocking out the silhouette of each console with primitive meshes, refined the geometry and surface flow, then set up materials, baked the GBA SP's flip-lid animation and exported each console as a standalone GLB file ready for the website.

01. Blocking out the console shape

I started by modelling the basic body of each console with simple primitive meshes — boxes for the Game Boy chassis, the GBA SP clamshell halves, and the PSP's flat slab. The goal was to lock in proportions, the screen cut-out, the D-pad area, and the GBA SP's hinge axis before adding any surface detail.

02. Refining geometry and modifiers

Once the basic forms were in place I used Subdivision Surface, Bevel, Weighted Normal and Smooth-by-Angle modifiers to round corners and tidy up shading without exploding the polycount. I also separated logical parts (lid, hinge, screen, buttons, UMD bay) so each part could be animated or recoloured independently in the browser.

03. Materials, animation and export

Finally I assigned PBR materials per part — matte plastic for the Game Boy, metallic paint for the GBA SP, piano-black gloss for the PSP — keyframed the GBA SP flip-lid as a single closing Action in the NLA editor, then exported each console as a glTF Binary (.glb) with "Apply Modifiers", "Always Sample Animations" and "+Y up" enabled so Three.js's GLTFLoader picks everything up cleanly.

What I used

Software and techniques

Blender

Used for modelling all three handheld consoles, editing mesh shapes, checking wireframes, applying modifiers, assigning PBR materials, keyframing the GBA SP flip animation and exporting GLB files.

HTML, CSS, JavaScript & Bootstrap 5

Used to build the multi-page site, lay out the dark museum visual style with CSS variables and a fluid Bootstrap grid, organise the exhibit cards, BGM player bar and control panels, and wire up all DOM events.

Three.js (r176) + Vite

Used to load each GLB through GLTFLoader, configure the scene with ambient + directional lights and a spot light, allow OrbitControls rotation and zoom, drive the GBA SP flip via AnimationMixer, toggle wireframe mode, change colours, switch camera presets and play background music.

Challenges

Problems I faced

01. The GBA SP hinge

Getting the flip-lid axis to rotate correctly required parenting the lid mesh to an empty positioned exactly on the hinge line. If the empty was even a few millimetres off, the lid would slide instead of pivot during animation playback.

02. Materials and surface finish

Each console needed a different feel — matte plastic, metallic paint, piano-black gloss. I had to tune metallic and roughness values per part and verify how they read under the museum's directional + spot lighting in Three.js, since Blender's preview never matches the browser exactly.

03. Polycount vs. detail

Some parts (the speaker grilles, the PSP's analog nub) needed enough geometry to look smooth, but heavy meshes hurt page load and frame rate on the three-card preview canvases. I balanced detail against file size by using Subdivision modifiers conservatively and removing unseen faces.

04. Exporting animation to GLB

Blender's glTF exporter silently dropped my flip animation the first time because the action wasn't pushed into an NLA strip. Once I pushed it down, ticked "NLA Tracks" + "Always Sample Animations" + "Apply Modifiers", the clip showed up correctly in gltf.animations[0].

05. Three.js integration

Wiring everything together — centring each model with a bounding-box recompute, switching screen meshes, normalising scale across three very differently-sized consoles, and making sure OrbitControls didn't fight with the canvas click raycaster — was the most time-consuming part.

06. Multi-page state

Because each console lives on its own HTML page, the BGM player would normally restart every time the user navigated. I solved this by persisting the player state (current track, time, volume, play/pause) to sessionStorage and rehydrating it on every page load.

Testing

What I tested and what changed

Functional testing was run after every milestone in Chrome (primary), Firefox and Edge, on both desktop (1920×1080) and a 1366×768 laptop screen. I also did a cognitive walkthrough of every page to check that a first-time visitor could understand the controls without help. The list below records the most important issues testing surfaced and the concrete fix that went into the codebase — each one is a testing-impact pair.

Canvas resize was overriding CSS

On Chrome, opening dev-tools changed the canvas to the wrong size and the 3D scene became blurry. Testing tied this to Three.js's renderer.setSize writing inline width/height. Fix: passed false as the third argument so CSS keeps controlling the canvas size.

Global BGM bar covered hero controls

After adding the persistent BGM bar, the Wireframe / Light / Flip / Rotate / Color buttons were partially hidden behind the navbar on every exhibit page. Fix: introduced a --bgm-bar-h CSS variable and subtracted it from every hero calc() so the layout stays correct at all breakpoints.

Flip button out of sync with camera presets

Clicking the "Open" camera preset on the GBA SP page moved the camera but didn't trigger the lid animation, and afterwards the Flip button toggled in the wrong direction. Fix: added a flipState field to camera presets and a getFlipState() callback so the button reads the true state from state.isFlipOpen instead of a local closure.

First flip click played the wrong way

Because the GBA SP GLB loads in the open pose at frame 0, the first click of the Flip button should close it. The original code started with isOpen = false so the first click tried to re-open an already-open lid. Fix: initial isOpen = true, matching the loaded pose.

Sub-path deployment broke all assets

Testing the first build on the Sussex ITS server gave an unstyled page — every CSS / JS / model / BGM 404'd because Vite produced absolute paths like /assets/.... Fix: set Vite base: './' and stripped the leading / from runtime asset paths in consoles.js so the build works under any sub-path.

Japanese BGM filenames were unsafe in URLs

The three BGM files use Japanese characters, spaces and parentheses. Although modern browsers usually encode these automatically, testing on a fresh profile occasionally failed the fetch. Fix: wrapped every BGM URL with encodeURI(...) before assigning it to the <audio> element.

Deeper understanding

Going beyond the lab tutorials

This section is my Deeper Understanding statement — features and techniques that go beyond what was covered in the Three.js labs, and how testing feedback shaped them.

MVC architecture

The codebase is split into model/ (data only — consoles.js), view/ (Three.js scene, loader, animations, post-processing) and controller/ (UI events, raycaster interaction, BGM player). No labs covered this separation; I designed it so future consoles can be added by editing consoles.js alone.

GSAP + AnimationMixer

Camera transitions and button-press feedback use GSAP (not taught in labs) for eased tweens; the GBA SP flip-lid uses Three.js's AnimationMixer to play a real GLTF clip embedded in Blender, with timeScale = ±1 letting one closing animation play forwards and backwards.

Multi-page Vite build

The lab tutorials used single-file demos. This project is a 5-page Vite build (rollupOptions.input with one entry per page), with base: './' so the produced dist/ can drop into any sub-path — discovered the hard way through deployment testing.

Persistent BGM player

A singleton player keeps state in sessionStorage (track index, time, volume, play/pause) so BGM continues across page navigation despite this being a multi-page site, not an SPA. Custom bgm:change events propagate state to the UI.

Bootstrap 5 + design system

Beyond using Bootstrap's grid, I built a custom design system on top: CSS variables for the museum palette, custom button classes (.btn-hero-ctrl, .btn-camera-preset), sticky-stacked top bars (BGM + navbar) and a fully responsive hero region.

Post-processing pipeline

EffectComposer wraps the renderer with a UnrealBloomPass so the bright parts of each console (LED, screen reflections) glow softly, and an OutputPass handles colour-space conversion. Configured per scene rather than globally so the bloom doesn't blow out the matte Game Boy.

Final structure

How the website is organised

The site has one landing page, three dedicated console exhibit pages and one about page. Splitting each console into its own page makes the project easier to navigate than one long comparison page, and lets each viewer be tuned independently (camera presets, flip controls, console-specific BGM).

Home screen

A landing page introducing Handheld Heritage with three cards — each card embeds a live 3D preview of one console (satisfying the "more than one model on a single page" requirement).

Exhibit pages

One page per console (Game Boy / GBA SP / PSP) with a full-screen 3D viewer, scene controls (wireframe, light, rotate, colour, flip), camera preset buttons and a specifications table.
